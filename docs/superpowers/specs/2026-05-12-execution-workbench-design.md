# 执行工作台优先的协作系统改造设计

- 日期：2026-05-12
- 范围：操作体验审查、流程语义优化、页面视觉优化
- 目标系统：`backend/` + `frontend/` 组成的任务/Bug 协作管理系统

## 1. 背景与目标

当前系统已经具备基础可用性：
- 后端有角色、权限、状态流转、历史记录和通知触发点
- 前端有 Dashboard、Projects、Tasks、Bugs、Users 等主要页面
- Task 与 Bug 已形成基本协作链路

但从实际使用体验看，系统目前更像一个“可管理流程的后台”，而不是一个“让执行人员一进来就能马上开工的工作台”。

本次改造目标不是推翻现有架构，而是在现有代码基础上完成以下三件事：

1. 让用户进入系统后立即知道“现在该做什么”
2. 让 Task / Bug 流程语义与用户感知更加一致
3. 在不重做设计风格的前提下，让页面更专业、更稳定、更有层级

## 2. 设计结论

本次改造采用“执行工作台优先，管理信息下沉”的方向。

设计原则如下：
- 首页优先展示当前用户的下一步动作，而不是先展示全局统计
- Tasks / Bugs 页面默认先展示与当前用户最相关的数据
- 详情区域优先展示当前角色在当前状态下可执行的动作
- PM / 组长仍保留全局管理视角，但不再作为所有角色的第一视觉重心
- 视觉优化以提升效率和信息层级为目标，不做换皮式重设计

## 3. 信息架构调整

### 3.1 Dashboard 从统计页改为工作台

`frontend/src/views/Dashboard.vue` 需要从“统计卡片 + 最近记录”的结构，调整为“行动区优先、总览区次之”的结构。

建议的新结构：

1. 顶部：当前行动区
2. 中部：与我相关的任务 / Bug 快速入口
3. 下部：项目总览、最近记录、全局数据

顶部行动区按角色动态变化，但结构保持统一：
- 开发：待开始、开发中、被打回、我的 Bug
- 测试：待测试、待验证、我创建的 Bug、我负责的验证项
- PM：待分配、测试中、待关闭、风险项
- 组长：本组待处理、本组进行中、待分配/待接手、异常项

这里的重点不是展示更多数字，而是把“下一步最可能要点击的内容”放到最前面，并确保卡片点击后能直接进入带条件的目标页面。

### 3.2 Tasks 页改为执行视角优先

`frontend/src/views/Tasks.vue` 当前更像总览页：先给一排筛选，再给看板/列表。改造后需要把“视角切换”放在筛选前面。

建议结构：
- 第一层：主视角切换
  - 我的待办
  - 我的参与
  - 全部任务
- 第二层：高级筛选
  - 项目
  - 状态
  - 优先级
- 第三层：内容展示（看板 / 列表）

默认主视角应按角色决定：
- 开发、测试默认进入“我的待办”
- 组长默认进入“我的参与”或“本组相关”
- PM 默认进入“全部任务”但保留行动入口快捷视图

看板和列表都保留，但不再让“全量数据 + 手动筛选”成为默认路径。

### 3.3 Bugs 页改为角色目标驱动

`frontend/src/views/Bugs.vue` 不能只是 Tasks 页的同构版本，因为 Bug 的处理目标不同。

建议结构：
- 第一层：主视角切换
  - 待我修复
  - 待我验证
  - 我创建的
  - 全部 Bug
- 第二层：高级筛选
  - 关联任务
  - 状态
  - 严重程度
- 第三层：内容展示

默认视角按角色切换：
- 开发优先看“待我修复”
- 测试优先看“待我验证”或“我创建的”
- 测试组长优先看“待验证”
- PM 看“全部 Bug”但保留风险聚合入口

### 3.4 导航语义调整

保留现有导航：`Dashboard / Projects / Tasks / Bugs / Users`。

但语义应当变化：
- Dashboard：工作台，不再只是统计页
- Projects：管理视图、归属和整体信息
- Tasks：任务执行与流转
- Bugs：问题修复与验证流转
- Users：仅 PM 的管理入口

## 4. 交互流程设计

### 4.1 首页卡片必须可行动

Dashboard 中的行动卡片和摘要卡片不能只是展示数字，必须支持直接跳转到具体上下文。

例如：
- 点击“待我处理 6 项”进入 Tasks 页并自动切换到“我的待办”
- 点击“待我验证 3 个 Bug”进入 Bugs 页并带上对应视角/筛选
- 点击“被打回 2 项”直接进入带 `rejected` 条件的任务视图

系统应替用户预先完成第一步筛选，减少重复操作。

### 4.2 详情抽屉改为动作导向

当前 `TaskDetailDrawer` 和 `BugDetailDrawer` 的方向是对的，但需要进一步收敛为动作导向界面。

建议详情抽屉分区如下：
1. 头部摘要：标题、状态、优先级/严重程度、核心责任人
2. 下一步动作区：当前角色真正可执行的操作按钮
3. 基础信息区：描述、关联项目/任务、截止时间等
4. 历史记录区：状态变化与备注
5. 协作补充区：必要备注或关联信息

关键规则：
- 只显示当前角色在当前状态下可以执行的动作
- 不提供与当前角色无关或无权限的误导性入口
- 状态变更按钮文案要贴近用户理解，而不是只暴露底层状态名

### 4.3 角色默认视图应由系统决定

当前系统已有角色鉴权，但“默认看什么”仍主要靠用户自己判断。

改造后，页面默认状态应当由角色自动决定，包括：
- 首页行动卡片内容
- Tasks/Bugs 页默认主视角
- 详情页优先展示的动作
- 快速入口中的文案重点

统一结构 + 差异化默认值，是这次交互优化的核心方式。这样既不打散系统一致性，也能显著降低执行用户的认知负担。

### 4.4 筛选交互从“裸控件堆叠”改为“主视角 + 高级筛选”

当前 `Tasks.vue` 和 `Bugs.vue` 顶部筛选条直接堆叠多个 `n-select`，属于典型后台风格，但对一线用户不够直接。

改造原则：
- 高频入口前置：先给“我的待办/待修复/待验证”等主视角
- 低频条件后置：项目、优先级、严重程度等保留在高级筛选
- 刷新按钮保留，但不作为主路径

## 5. 流程语义与逻辑修正

### 5.1 Task 状态语义需要变得可解释

当前 `backend/handlers/task.go` 中存在 `developed -> pending_test` 的自动二次流转。逻辑上可运行，但用户感知上不透明。

设计要求：
- 前端将“开发完成并进入测试池”表达为一个完整且可理解的动作
- 历史记录中明确标注系统自动加入测试池的原因
- 按钮文案、提示文案和状态文案保持一致

是否保留自动二次流转可以在实现阶段决定，但无论采用哪种方式，都必须保证用户对状态变化的理解是一致的。

### 5.2 Bug 流程语义必须统一

当前模型定义了 `pending_verify`，但现有处理流程中 `fixed` 与 `closed/reopened` 的关系不够一致，容易形成“状态定义存在，但流程未严格使用”的问题。

设计要求是统一为单一语义流程：
- `assigned -> fixing -> fixed -> pending_verify -> closed/reopened`

如果决定保留“待验证”这个产品概念，就必须在后端状态机、前端视图、详情动作和历史记录里完整体现，不能让它只停留在枚举或文案层。

### 5.3 页面动作入口必须与权限规则对齐

后端当前已有 `RoleRequired` 和状态流转校验，但前端也必须同步收口：
- 用户打开详情时，应优先看到自己能做的下一步
- 不应让用户先点击、再在失败后得知不能操作
- 页面上可见但不可执行的入口要尽量减少

这里的产品标准是“默认帮助正确操作”，而不是“允许点错后再报错”。

### 5.4 首页统计口径需要区分“我的行动”和“全局参考”

Dashboard 如果既承担工作台，又混入大量全局统计，用户容易误认为首页所有数字都与自己直接相关。

因此必须分为两组：
- 我的行动数字：直接关联下一步动作
- 全局参考数字：帮助理解整体进展，但不作为当前主任务

视觉上也应区分这两类信息，避免放在同一层级上造成误解。

## 6. 视觉设计方向

本次视觉优化不做大改风格，采用“轻量专业化”的方向。

目标不是让页面更花，而是让页面更像真实团队工作台：重点更清楚、反馈更明确、结构更统一。

### 6.1 强化信息层级

现有页面的问题不是缺少视觉元素，而是重点区域与次级区域之间的差异不够明显。

优化原则：
- 主行动区最突出
- 次级摘要区弱化边框和装饰
- 避免所有卡片同等强调
- 模块大小与视觉重量应与业务重要性一致

### 6.2 统一设计语言

Tasks、Bugs、Dashboard 需要共享一致的视觉规范：
- 圆角体系
- 间距节奏
- 标题层级
- 筛选栏样式
- 列表容器样式
- 详情抽屉分区样式

统一之后，系统会更像一个完整产品，而不是多个相近页面的拼接。

### 6.3 状态颜色服务业务语义

建议统一状态颜色语义：
- 待处理 / 待分配 / 待验证：蓝色或中性偏蓝
- 进行中：橙色
- 完成 / 已通过：绿色
- 打回 / 重开 / 风险：红色
- 已关闭：灰色

颜色使用必须稳定，保证用户可以凭第一眼完成风险判断与优先级判断。

### 6.4 增强可点击感与反馈感

对以下元素加强交互反馈：
- 行动卡片
- 列表行
- 详情入口按钮
- 状态动作按钮
- 快速筛选项

需要提供明确的 hover、active、focus 反馈，让页面从“展示后台”变成“可操作工作台”。

### 6.5 兼顾桌面场景下的响应式与密度控制

当前系统明显以桌面端为主，因此改造重点是：
- 宽屏下保持舒展，不让内容过于稀薄
- 中等宽度下避免筛选栏拥挤换行失控
- 列表与卡片的信息密度保持在高效但不压迫的区间

## 7. 实施优先级

本次改造按以下顺序推进：

### P1：操作体验优先
- Dashboard 顶部行动区重构
- Tasks/Bugs 默认主视角切换
- 高价值快捷入口设计
- 详情页动作导向化

### P2：流程语义修正
- Task “开发完成 -> 进入测试池”语义统一
- Bug “待验证”流程统一
- 页面动作与状态权限收口

### P3：视觉统一与精修
- 统一卡片、筛选栏、抽屉、列表样式
- 调整状态色、层级、间距、按钮反馈
- 强化 hover / active / selected 感知

## 8. 非目标

为了控制范围，本次改造不包含以下内容：
- 大规模后端架构重写
- 新增复杂业务子系统
- 完全不同的品牌风格重设计
- 与操作体验/流程语义/工作台视觉无关的全面重构

## 9. 测试与验收策略

### 9.1 后端验证

优先补充以下测试：
- Task 状态流转测试
- Bug 状态流转测试
- 角色权限测试

重点验证状态图、角色限制和历史记录写入是否一致。

### 9.2 前端验证

需要验证以下核心行为：
- 不同角色进入首页时，默认行动区内容是否正确
- Tasks / Bugs 页面默认主视角是否符合角色预期
- 详情页是否只展示当前可执行动作
- 行动卡片跳转后是否正确带入目标上下文

### 9.3 人工验收标准

改造完成后，至少应满足以下结果：
- 开发进入系统后，能在几秒内定位当前待办
- 测试能快速区分“待测 / 待验证 / 我提交的 Bug”
- PM 仍能保留必要的项目总览与风险视角
- 用户不需要频繁先筛选再决定下一步做什么
- 页面整体视觉更成熟统一，但不影响现有使用习惯

## 10. 预期结果

按本设计完成后，系统将从“能管理流程”提升为“能支撑日常协作”：
- 执行人员更快开工
- 管理人员仍保留总览能力
- Task / Bug 状态更容易理解
- 页面更专业、更统一、更具有行动导向
